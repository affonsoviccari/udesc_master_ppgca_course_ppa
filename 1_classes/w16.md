# Programação com OpenMP

```#pragma``` diretiva para incluir argumentos para o compilador
- Não é feito para memória distribuída.
- Não garante a melhor utilização da memória compartilhada.
- Não verifica dependências ou deadlocks.
- Não há *preempção de threads\**
> Preempção de threads é o mecanismo do sistema operacional que interrompe assincronamente uma thread em execução para dar lugar a outra, geralmente de maior prioridade.

O código deve ser segregado, uma vez que o OpenMP não verifica dependências e deadlock, então deve ser controlado as partes que podem ser paralelizadas.

```C
#include <omp.h>

---

gcc nome_do_arquivo.c -fopenmp -o nome_do_arquivo
```

Com OpenMP, [test1.c](../4_repos/c_cpp/4_openmp/test1.c).

```c
#include <stdio.h>
#include <omp.h>

int main (void) {
    int id;
    #pragma omp parallel
    {
        id = omp_get_thread_num();
        printf("UDESC PPA 2026-1 = %d\n", id);
    } // join

    return 0;
}

---

gcc -Wall main.c -o out -fopenmp
```

`#pragma omp parallel` aplica a compilação em paralelo somente para a próxima instrução de código.

`int id` possui escopo privado ou compartilhado? Por padrão é privada, porém pode-se passar o argumento `shared()`, `private()` ou `firstprivate()`.
- `firstprivate`, apenas recebe o valor default de fora do escopo.

OpenMP substitui `thread_create()`, `thread_join()` e `thread_exit()`.

`#pragma omp parallel num_treads(N)`, `num_threads(N)` coloca um número N de threads para trabalhar em paralelo no bloco `#pragma`.

## Clause List
`#pragma omp master`
`#pragma omp single`, somente uma thread executa a instrução dentro do bloco.
`#pragma omp critical`, como se fosse sequencial, ou mutex, a instrução abaixo o código será feito de forma sequencial.

## Sections
`#pragma omp sections nowait` que separa o código em seções `#pragma omp section`.

## Tasks
Uma tarefa, criada com `#pragma omp task`. Em memória é craida uma hierarquia de tarefas, sabendo o sequenciamento das tarefas.

O mapeamento de carga deve ser dinâmico, de N threads para M tasks.

## Vetorial
`#pragma omp for`, decompõe o for, em um paralelismo de dados e não de sessões. Particiona de forma estática e igualitária entre threads.

### omp_schedule.cpp
`NUM_THREADS` é 4 e existem políticas de escalonamento, com um vetor gigantesco, é particionado com o número de threads.
- Tamanho/fatia == Chunk (N:M).

Cada thread processa e dorme (`sleep_EVERY_N` por ms), o sleep serve para liberar processamento e também se o tempo de processamento for muito menor que o tempo do sleep, também sincroniza as threads.

- `nStatic1[NUM_LOOPS], nStaticN[NUM_LOOPS]` - Cada célula do vetor recebe uma thread para excecução. Nesse caso há disputa para cache, e obrigatoriamente utilizou todas as 4 threads.
- `nDynamic1[NUM_LOOPS], nDynamicN[NUM_LOOPS]` - Separado por chunks definidos.
- `nGuided[NUM_LOOPS]` - Guiado pelo runtime, começa grande e diminui o problema, diminuindo a probabilidade de balanceamento de carga.

> O escalonamento utilizando `nStatic1`, carrega todo o conjunto no L3 da cache, já o nStaticN, carrega na L3 o conjunto dinâmico.

```
     []     [] L1
    [  ]   [  ] L2 = L3/2
[                 ] L3
```

A troca de informação entre L1, L2 e L3 são muito rápidas, porém o carregamento de memória na cache L3 é muito lento, outra grandeza.

O processo de segregar e particionar o vetor está na etapa de Aglomeração no PCAM.
